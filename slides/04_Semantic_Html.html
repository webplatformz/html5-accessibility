Semantic HTML != Meaningful

Semantic & meanigful HTML

  • DOM Structure
  • No excessive/useless tags & attributes
  • No omission of relevant tags & attributes
  • Do not abuse tags for style & layout
Screenreaders evaluate (mostly only) the structure of tags & attributes

Document

You MUST provide a valid context for the document

  • <!doctype html>
  • <html lang="en">
  • <meta charset="utf-8">
Changing this after initial load might not work properly

noscript

Forget about <noscript> - that is about javascript not accessibility !

Modern Screenreaders should be able to deal with Javascript

noscript ... BUT

A reader-friendly, clutter-free view is mostly appreciated by people with disabilities ;)

K.I.S.S.

Heading

Some Screanreaders might analyse & extract headings in order to generate an index of the page

  • Exactly one <h1> per page
  • Strict order (h1 > h2 > h3 ...) is mandatory
  • Do not depend on default browser styles
  • No invalid nesting <h1><h2>...
Strict order is also relevant for SEO.
Google will punish you if you violate this !

Heading

  • Do not abuse existing headings(-styles) for "emphasis"
  • Do not (ab)use <bold>, <strong>, <em>, ... for titles
  • Do not combine with <bold>, <strong>, <em>, ...
Remember: Headings are probably announced as such

Navigation

  • Ideally at most one <nav> per page
  • At most one <nav> inside container (<main>, <aside>, <footer>, <header>)
  • Nesting <nav> is valid, but not recommended
  • Maybe use aria-label="Primary Navigation" to differentiate
Too many navigation(-levels) can be very confusing due to announcing

Navigation: "Skip-List/Navigation"

  • At most one per page
  • First element inside <body>
  • Do not use display:none or visibility:hidden
                    
                        <nav class="skip-navigation">...</nav>
                        .skip-navigation {
                            position: absolute;
                            top: -100000px;
                            left: -100000px;
                        }

Layout - Example page structure

Page-level <header>, <main>, <aside>, <footer>

                    
                        <body>
                            <header> ... # first
                            <main> ...   # same level
                            <aside> ...  # same level
                            <footer> ... # last
                        </body>

No wrapping/enclosing <div> !

Layout - Example section structure

Section-level <header>, <main>, <aside>, <footer>

                    
                        <section>
                            <header> ... # first
                            <main> ...   # same level
                            <footer> ... # last
                        </section>

No wrapping/enclosing <div> !

Layout - Example article structure

Article-level <header>, <main>, <aside>, <footer>

                    
                        <article>
                            <header> ... # first
                            <main> ...   # same level
                            <footer> ... # last
                        </article>

No wrapping/enclosing <div> !

Layout - General structure

Don't use <header>, <main>, <aside>, <footer> for anything else

Do not nest <header>, <main>, <aside>, <footer>

Only use if really necessary => complexity !

Section

Like <div>, BUT semantic unit grouping related things

  • Top-level element (e.g. child of <body>, <main>, ...)
  • Can be nested
  • Avoid too many nesting levels
  • Probably announced by Screenreader

Article

Semantic unit containing everything that is related to it

  • It may contain multiple tags like <p>, <img>, ...
  • Never ever nest <article>
  • Announced by Screenreader

Div

"Ok" for grouping and structuring (anywhere)

  • Mostly ignored
  • Could be anything
  • Not (content) meaningful
  • Never announced or highlighted by Screenreader

Lists

  • Ranking: <ol> + <li>
  • Order: <ul> + <li>
  • Map or Key/Value-Pair: <dd> + <dl>
  • Keep the nesting-level as small as possible

Do not use <table> for simple Key/Value Pairs

Form

  • Do not enforce a pointing device (e.g. mouse) upon the user
  • Always provide an alternative way for input
  • Do not clutter the form with irrelevant (semantic) elements

Form

  • Avoid fancy UI component libraries/frameworks (unless truely optimized for accessibility)
  • Use mostly native HTML 5 elements

Form - Must Do

Proper correlation between form elements

  • for=
  • id= and/or name=
  • Group related form inputs: <formfield>>
  • Purpose: role="search"
Most role= for form elements have become redundant

Form - "natural reading flow"

Enable a natural reading flow

  1. <label>
  2. <input|select|textarea>

Also avoid any clutter around/inbetween label and input|select|textarea

Form - Attributes

Use "new" HTML 5 attributes and values

  • Input type: type= (url, tel, email, date, datetime, month, time, color, range, ...)
  • Input/Value constraints: required, multiple, readonly, minlength=, maxlength=, min=, max=, ...
  • Input example: placeholder=
  • Avoid pattern="your-regex"
There is no alt= explaining the input format

Form - Microformat

Do not combine "Microformat" and <form>

Microformat intends to give extra meaning to content

  • itemprop="street-address"
  • itemprop="postal-code"
  • rel="author"

Table

This tag is meant for tabular-data content only !

  • Should have <thead>, <tbody>, <tfoot>
  • Header cells must use <th>

Table - Screenreaders

Put tabular structures into CSS grid/flexbox !

  • Have become very smart
  • May sniff-out tabular structures
  • Tabular-structures will totally mess up reading mode

Table - Reading semantics

  • Read by column: <tr><th scope="col">...
  • Read by row: <tr><th scope="row">...
  • Avoid multiple header/footer rows/columns
  • Avoid colspan= and rowspan=

Anchors, Buttons

Quick access through "tab"- and "enter"-key by default

  • Always announced by Screenreader
  • No empty title= or alt=
  • Interaction/Form: <button>
  • Navigation: <a>

Anchors, Buttons - Animations

"Guide" the user to the newly visible content-section

This is easier said than done (e.g. VoiceOver)

Media

  • Avoid external libraries for audio/video players
  • Do not/never autoplay
  • Use native <audio> and <video>
  • Avoid <object>, <embed>, <canvas>
  • Supply <track> for <audio> and <video>

Media - image

This tag is meant for content only

  • Use <img> for photos only
  • Always provide alt=
  • Do not use <img> for theme, design and/or background images

Media - figure

This tag is meant for content only

  • Use <figure> for graphics, diagrams, illustrations
  • Always use an appropriate <figcaption>
  • Do not use <caption> (for tables only!)
  • Do not use <figure> for theme, design and/or background images

Media - figure

                    
                        <figure>
                            <img src= alt= > # image
                            <figcaption>...</figcaption> # description
                        </figure>

alt= must still be set on <img> !

Media - picture

Native responsiveness - No more library required ! :)

                    
                        <picture>
                            <source media=... srcset=... > # source 1
                            ...
                            <source media=... srcset=... > # source N
                            <img ... > # fallback
                        </picture>

alt= must be set on <img> !
